# Iteration 14: Text Output Overhaul

## Goal

Redesign `--format text` output across all commands to be scannable and human-friendly. The current text format (especially for `find`) dumps verbose `name=X, type=Y, value=Z` triplets that are nearly unusable for quick scanning.

## Root Cause

Text output is generated by jaq (jq) filters matched against JSON key signatures. This works for simple shapes (`PropertyInfo`, `TagSummary`) but breaks for:

1. **`FileObject` (from `find`)** — Key signature varies with `--fields`, so no filter matches. Falls through to `format_object_generic`, which renders nested objects as `name=value` comma-joined pairs on a single line.
2. **`FindTaskInfo`**, **`ContentMatch`** — No dedicated filters; fall through to generic `key=value`.
3. **Mutation results** (`SetPropertyResult`, `RemoveTagResult`, etc.) — Also no filters.

The fix: dynamically compose jaq filter strings based on which fields are present, plus add static filters for the remaining unhandled shapes.

## Approach: Dynamic jaq Filter Composition

Instead of a Rust-side formatter, `FileObject` gets a **dynamically composed jaq filter**. When `format_value_as_text` detects a `FileObject` (has `file` + `modified` keys), it inspects which optional fields are present and assembles a jaq filter from reusable fragments:

```rust
fn build_file_object_filter(keys: &BTreeSet<&str>) -> String {
    let mut filter = r#""\(.file)  (\(.modified))""#.to_string();
    if keys.contains("properties") {
        filter += r#" + "\n" + (.properties | map("  " + ...) | join("\n"))"#;
    }
    if keys.contains("tags") { /* append tags fragment */ }
    if keys.contains("sections") { /* append sections fragment */ }
    // ... tasks, matches, links
    filter
}
```

This keeps the single rendering path through jaq — no special Rust-side formatting code needed. The filter is assembled at runtime but executed by jaq like all the others.

## Design Decisions

- **Dynamic filter composition**: Assemble jaq filters at runtime from fragments based on which `FileObject` fields are present — avoids 2^6 static filter variants, keeps single jaq rendering path
- **FileObject detection**: Check for `file` + `modified` keys in `lookup_filter` / `format_value_as_text`
- **Array separators**: Blank line between `FileObject` entries; tight `\n` for single-line items
- **Pipe-friendly**: No ANSI colors. 2-space indentation. Grep-friendly file path headers
- **JSON output unchanged**: Only `--format text` rendering changes
